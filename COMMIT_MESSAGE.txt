Fix Ollama client, session mgmt, and Qdrant point IDs for working ingestion/retrieval

## Changes

### Backend Fixes
- Replaced global ollama instance with per-call Client(host=settings.OLLAMA_BASE_URL)
  in ingest.py and retrieval.py to respect configured Ollama endpoint
- Implemented per-call DB sessions with try/finally in ingest_document to prevent
  connection leaks and stale session issues
- Fixed Qdrant point ID validation: changed chunk_id from string concat 
  (e.g., "uuid_chunk_0") to UUID5-based IDs (uuid.uuid5(doc_id, f"chunk-{idx}"))
  so Qdrant accepts point IDs as valid UUIDs
- Improved JSON extraction in entity parsing with fallback regex for robust LLM output

### DevOps & Configuration
- Updated .gitignore to exclude all venv directories (venv/, venv_new/, .venv/, .venv2/)
  and Python build artifacts (*.egg-info/, dist/, build/, .pytest_cache/, .coverage)

### Testing & Verification
- End-to-end ingestion and retrieval now functional
- Confirmed librarian can ingest PDFs and retrieve/synthesize answers using hybrid RAG
- Backend health check: api/qdrant/postgres/ollama all available
- Frontend (Next.js) and backend (FastAPI) successfully running locally

## Known Issues
- File upload system: handling of failed/duplicate files needs improvement
- Knowledge graph entity extraction: accuracy and relationship creation need refinement
- Docker-to-Ollama connectivity: Windows firewall blocks container bridge to host Ollama
  (workaround: run backend locally)

## Tested Workflow
1. Upload PDF → ingest to PostgreSQL chunks + Qdrant vectors + graph nodes
2. Query librarian → vector search + graph expansion + LLM synthesis
3. Result: accurate answers with citations and relationship context
